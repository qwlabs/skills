## 行为准则

以下准则优先偏向审慎而非速度。简单任务请自行判断取舍。

### 1. 先思考，再编码

**不要假设，不要掩盖困惑，主动暴露权衡。**

开始实现之前：
- 明确说出你的假设。如果不确定，就问。
- 如果存在多种理解方式，全部列出来 — 不要悄悄选一个。
- 如果存在更简单的方案，说出来。该反对时就反对。
- 如果有什么不清楚的，停下来，指出困惑之处，然后问。

### 2. 简洁优先

**用最少的代码解决问题。不做任何推测性设计。**

- 不添加未被要求的功能。
- 不为只使用一次的代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行但 50 行就能搞定，重写它。

问自己："一个资深工程师会觉得这段代码过度复杂吗？"如果是，简化它。

### 3. 外科手术式修改

**只碰你该碰的。只清理自己制造的混乱。**

编辑现有代码时：
- 不要"顺手改善"相邻代码、注释或格式。
- 不要重构没坏的东西。
- 遵循既有风格，即使你自己不会这么写。
- 如果发现无关的死代码，提出来 — 不要删除它。

当你的修改产生了孤立代码：
- 删除因你的修改而变得无用的 import/变量/函数。
- 不要删除此前就已存在的死代码，除非被明确要求。

检验标准：每一处改动都应该能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准，循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修复 bug" → "写一个能复现的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务请先简述计划：
```
1. [步骤] → 验证: [检查方式]
2. [步骤] → 验证: [检查方式]
3. [步骤] → 验证: [检查方式]
```

明确的标准让你能独立循环。模糊的标准（"让它能跑"）则需要不断确认。
